## **audio chunk (16000Hz + 20ms chunk)**

**Audio chunk**는 음성 데이터를 **작은 단위로 잘라서 보내는 것**을 의미합니다.

실시간 음성 시스템에서는 전체 파일을 보내지 않고 **짧은 시간 단위의 데이터 조각**을 계속 전송합니다.

---

# 1️⃣ audio chunk 개념

마이크는 실제로 이렇게 데이터를 생성합니다.

```
audio stream
```

이 스트림을 일정 길이로 나누면 **audio chunk**가 됩니다.

예

```
전체 음성
────────────────────────────

chunk1  chunk2  chunk3  chunk4
```

각 chunk는 보통 **10~100ms 길이**입니다.

---

# 2️⃣ 왜 chunk로 나누는가

만약 파일로 보내면

```
5초 녹음
↓
파일 업로드
↓
STT
```

문제

- 응답이 늦음
- 실시간 대화 불가능

그래서 이렇게 합니다.

```
chunk1 → 서버
chunk2 → 서버
chunk3 → 서버
```

그러면 서버는 **말하는 동안 STT를 시작할 수 있습니다.**

---

# 3️⃣ 실제 데이터 예

예를 들어

```
sample rate = 16000 Hz
```

즉

```
1초 = 16000 samples
```

만약 **20ms chunk**라면

```
0.02 × 16000 = 320 samples
```

그래서 하나의 chunk는

```
320 samples
```

입니다.

---

# 4️⃣ 라즈베리에서 chunk 생성

Python에서는 마이크 버퍼를 그대로 chunk로 사용합니다.

예

```
importsounddeviceassd

defaudio_callback(indata,frames,time,status):
chunk=indata.tobytes()

send_to_server(chunk)

stream=sd.InputStream(
samplerate=16000,
channels=1,
callback=audio_callback
)

stream.start()
```

이 코드에서

```
indata
```

가 바로 **audio chunk**입니다.

흐름

# 6️⃣ 서버에서 처리

서버는 chunk를 계속 받아서 STT 엔진에 넣습니다.

```

```

이렇게 누적됩니다.

```

```

이렇게 **실시간 텍스트 생성**이 됩니다.

# 8️⃣ chunk 크기 추천

실제 음성 시스템에서 많이 사용하는 값

```
sample rate = 16000
channels = 1
chunk size = 320 ~ 1024 samples
```

시간 기준

```
20ms ~ 64ms
```

이 정도가 **지연과 안정성 균형이 좋습니다.**

## Stream

핵심 개념을 먼저 정리하면 다음과 같습니다.

> **chunk를 STT에 “매번 요청”하는 것이 아니라, 보통 “스트림(stream)”으로 계속 전달합니다.**

즉 HTTP처럼 **요청을 여러 번 보내는 방식**이 아니라

**연결 하나를 유지하면서 chunk를 계속 보내는 방식**입니다.

---

# 1️⃣ 두 가지 방식

음성 데이터를 STT에 보내는 방식은 크게 두 가지입니다.

## ① 파일 요청 방식 (비실시간)

```
라즈베리
 └ 녹음 (3초)

      ↓

HTTP 요청
 └ audio.wav 전송

      ↓

STT
 └ 텍스트 반환
```

문제

- 응답 느림
- 실시간 대화 불가능

그래서 AI 음성 시스템에서는 잘 안 씁니다.

---

## ② 스트리밍 방식 (실시간)

실제 음성 시스템은 이 방식입니다.

```
라즈베리
 └ WebSocket 연결

chunk1 →
chunk2 →
chunk3 →
chunk4 →

STT 서버
 └ 실시간 분석
```

즉

```
연결 1개
데이터 계속 전송
```

입니다.

---

# 2️⃣ 실제 흐름

사용자가

```
안녕하세요
```

라고 말하면

라즈베리에서는 이렇게 됩니다.

```
20ms audio
20ms audio
20ms audio
20ms audio
```

이 chunk들이

```
chunk1
chunk2
chunk3
chunk4
```

형태로 계속 서버로 전송됩니다.

---

# 3️⃣ 서버 내부 처리

STT 서버는 이렇게 처리합니다.

```
chunk1 수신
 ↓
buffer 저장

chunk2 수신
 ↓
buffer 누적

chunk3 수신
 ↓
STT 분석
```

그래서 중간에

```
안
안녕
안녕하세요
```

같은 **부분 결과(partial result)**도 나올 수 있습니다.

---

# 4️⃣ 코드 흐름 예시

라즈베리 (클라이언트)

```
whileTrue:

chunk=mic.read()

websocket.send(chunk)
```

---

서버

```
whileTrue:

chunk=websocket.receive()

audio_buffer.append(chunk)

text=stt(audio_buffer)
```

입니다.

## VAD ( Voice Activity Detection )

**VAD (Voice Activity Detection)** 는

오디오 신호에서 **“지금 사람이 말하고 있는지 / 아닌지”**를 판단하는 기술입니다.

즉 마이크 입력에서 **음성 구간만 골라내는 필터**라고 보면 됩니다.

---

# 1️⃣ 왜 VAD가 필요한가

마이크는 항상 소리를 듣습니다.

```
마이크 입력
```

하지만 실제로는 대부분이

```
무음
환경 소음
키보드 소리
```

입니다.

예

```
[무음][무음][무음][말][말][말][무음]
```

VAD는 이 중에서 **말하는 구간만 찾습니다.**

```
VAD 결과
```

```
      [말][말][말]
```

---

# 2️⃣ 음성 시스템에서의 위치

AI 음성 시스템에서는 보통 이런 순서로 동작합니다.

```
마이크
 ↓
Wake Word
 ↓
VAD
 ↓
STT
```

흐름을 보면

```
퍼비야
```

wake word 감지

그 다음

```
오늘 날씨 알려줘
```

이 구간을 **VAD가 감지**합니다.

---

# 3️⃣ VAD가 하는 일

VAD는 크게 두 가지를 판단합니다.

### 1️⃣ 음성 시작

```
무음 → 말 시작
```

### 2️⃣ 음성 종료

```
말 끝 → 무음
```

그래서 이런 결과를 만듭니다.

```
speech_start
speech_end
```

---

# 4️⃣ 예시

사용자가 말할 때

```
[무음][무음][무음][안][녕][하][세][요][무음]
```

VAD는 이렇게 판단합니다.

```
speech start → "안"
speech end → "요"
```

그래서 STT에는

```
안녕하세요
```

이 구간만 전달됩니다.

---

# 5️⃣ VAD가 없으면 생기는 문제

만약 VAD 없이 chunk를 계속 보내면

```
[무음][무음][무음][무음]
```

이것도 계속 STT에 보내게 됩니다.

문제

- 서버 비용 증가
- 인식 오류 증가
- 처리 지연

그래서 대부분 시스템은

```
말할 때만 STT
```

합니다.

---

# 6️⃣ 실제 음성 시스템 구조

AI 음성 디바이스는 보통 이렇게 구성됩니다.

됩니다.

## 왜 음성 시스템은 WebSocket을 사용하는가

음성은 **연속 데이터(stream)** 입니다.

그래서 일반 HTTP 요청 방식은 비효율적입니다.

### HTTP 방식

```
라즈베리
 └ 녹음 (3초)

      ↓ HTTP 요청

서버
 └ STT

      ↓ 응답

텍스트 반환
```

문제

- 녹음이 끝나야 STT 시작
- 응답 지연 발생

---

### WebSocket 방식

```
라즈베리
 └ WebSocket 연결 유지

chunk1 →
chunk2 →
chunk3 →
chunk4 →

STT 서버
 └ 실시간 분석
```

장점

- **실시간 스트리밍**
- **지연 감소**
- **양방향 통신**

그래서 대부분 음성 시스템은

```
audio streaming → WebSocket
```

을 사용합니다.

## STT latency를 줄이는 방법

좋은 음성 시스템은 **사용자가 말을 끝낸 뒤 1초 안에 응답**해야 합니다.

지연이 발생하는 주요 구간

```
음성 녹음
STT 처리
LLM 처리
TTS 생성
```

이를 줄이기 위한 방법이 있습니다.

---

### 방법 ① Streaming STT

chunk를 계속 보내면서 **말하는 동안 STT 실행**

```
audio chunk → STT
audio chunk → STT
audio chunk → STT
```

---

### 방법 ② Partial transcription

STT가 중간 결과를 생성

예

```
사용자: 안녕하세요

STT
안
안녕
안녕하세요
```

---

### 방법 ③ Endpoint detection

사용자가 말을 끝냈다는 것을 빠르게 판단

예

```
0.5초 무음
```

→ 발화 종료

## Endpoint Detection

# 1️⃣ Endpoint Detection이란

**Endpoint Detection**은

사용자가 **말을 시작한 시점과 끝낸 시점을 판단하는 기술**입니다.

특히 중요한 것은

```
speech end detection
```

입니다.

즉 시스템이

> **"사용자가 이제 말을 끝냈다"**

라고 판단하는 순간입니다.

---

# 2️⃣ 왜 Endpoint Detection이 필요한가

사용자가 이렇게 말한다고 가정합니다.

```
"오늘 날씨 알려줘"
```

문제는 **AI가 언제 STT를 실행해야 하는지**입니다.

만약 Endpoint Detection이 없다면

```
오늘 날씨 알려줘
(2초 기다림)
STT 실행
```

같이 됩니다.

이러면 시스템이 **느리게 느껴집니다.**

---

# 3️⃣ Endpoint Detection이 있는 경우

Endpoint Detection이 있으면

```
오늘 날씨 알려줘
↓
0.3~0.5초 무음 감지
↓
발화 종료 판단
↓
STT 실행
```

그래서

```
응답 속도
```

가 크게 개선됩니다.

---

# 4️⃣ 실제 동작 흐름

음성 데이터는 보통 이렇게 보입니다.

```
[무음][무음][안][녕][하][세][요][무음][무음]
```

Endpoint Detection은

```
speech start
speech end
```

를 판단합니다.

예

```
speech_start → "안"
speech_end → "요"
```

---

# 5️⃣ Endpoint Detection 알고리즘

대부분 다음 두 가지를 조합합니다.

### 1️⃣ VAD 기반

VAD가

```
speech
non-speech
```

를 판단합니다.

그리고

```
0.5초 이상 무음
```

이면 발화 종료로 판단합니다.

---

### 2️⃣ Energy 기반

오디오 신호의 **에너지(level)**를 분석합니다.

예

```
음성 → 에너지 높음
무음 → 에너지 낮음
```

이걸 이용해 발화 종료를 판단합니다.

---

```

```

```

```

정도가 됩니다.

Endpoint Detection은

> **사용자가 말을 끝냈다는 시점을 판단하는 기술**

입니다.

그래서 Voice Pipeline에서

```

```

순서로 사용됩니다.

## Barge-in

# 1️⃣ Barge-in이란

**Barge-in**은

> AI가 말하고 있는 중에도 사용자가 끼어들어 말할 수 있게 하는 기능입니다.

예

```
AI: 오늘 날씨는 맑고 22도이며...
사용자: 아니 내일 날씨!
```

이때 시스템이 해야 하는 일

```
1 AI 음성 중단
2 사용자 음성 감지
3 새로운 STT 시작
```

이것이 **Barge-in**입니다.

---

# 2️⃣ Barge-in이 없는 시스템

Barge-in이 없으면 이런 상황이 됩니다.

```
AI: 오늘 날씨는 맑고 22도이며...

사용자: 아니 내일 날씨!

AI: ...풍속은 초속 2미터입니다.
```

즉

```
사용자 입력 무시
```

됩니다.

그래서 대화가 매우 **부자연스럽습니다.**

---

# 3️⃣ Barge-in이 있는 시스템

Barge-in이 있으면 이렇게 됩니다.

```
AI: 오늘 날씨는 맑고 22도이며...

사용자: 아니 내일 날씨!

AI: (음성 중단)

STT → "내일 날씨"

AI: 내일은 비가 올 예정입니다.
```

이 방식이 **Alexa / Google Assistant 같은 시스템**입니다.

---

# 4️⃣ Barge-in 구현 핵심

Barge-in을 구현하려면 시스템이 동시에 두 가지를 해야 합니다.

```
1 AI 음성 재생
2 사용자 음성 감지
```

즉

```
speaker output
microphone listening
```

을 **동시에 실행**해야 합니다.

# 10️⃣ 핵심 정리

Barge-in은

> **AI가 말하는 중에도 사용자가 끼어들 수 있게 하는 기술**

```
마이크 항상 ON

TTS interrupt
```

구조가 필요합니다.

## Turn Detection

**Turn Detection**은 음성 대화 시스템에서

**“지금 누구의 차례인지”**를 판단하는 기술입니다.

즉,

> **사용자가 말하는 차례인지, AI가 말하는 차례인지 결정하는 시스템**

입니다.

이 개념이 있어야 **자연스러운 대화 흐름**이 만들어집니다.

---

# 1️⃣ Turn Detection의 기본 개념

대화는 항상 **턴(turn)**으로 이루어집니다.

예

```
사용자: 오늘 날씨 알려줘
AI: 오늘 서울은 맑고 22도입니다
사용자: 내일은?
AI: 내일은 비가 올 예정입니다
```

이때 시스템은 항상 판단해야 합니다.

```
지금 사용자 턴인가?
지금 AI 턴인가?
```

이 판단을 **Turn Detection**이 합니다.

---

# 2️⃣ Turn Detection이 필요한 이유

음성 인터페이스에서는 **말이 끝났는지 판단하는 것이 어렵습니다.**

예

```
오늘 날씨 알려줘... 음...
내일도 알려줘
```

여기서

```
"오늘 날씨 알려줘"
```

에서 AI가 응답하면 **대화가 깨집니다.**

그래서 시스템은 판단해야 합니다.

```
사용자가 아직 말하는 중인가?
정말 끝났는가?
```

---

# 3️⃣ Turn Detection이 사용하는 신호

보통 여러 신호를 조합합니다.

### ① VAD (음성 여부)

```
speech / silence
```

### ② Endpoint Detection

```
0.5초 무음 → 발화 종료
```

### ③ 문장 구조

STT 결과를 보고 판단

예

```
"오늘 날씨 알려줘"
```

→ 질문 완성

### ④ 억양 (prosody)

사람은 질문 끝에 억양이 올라갑니다.

예

```
내일은?
```

---

# 4️⃣ 실제 Voice Pipeline에서의 위치

Voice Pipeline에 Turn Detection을 넣으면 이렇게 됩니다.

```
Wake Word
 ↓
VAD
 ↓
Endpoint Detection
 ↓
Turn Detection
 ↓
STT
 ↓
LLM
 ↓
TTS
```

Turn Detection이

```
지금 AI가 말할 차례
```

라고 판단하면

```
LLM 호출
```

합니다.

---

# 5️⃣ Turn Detection 예시

사용자 발화

```
오늘 날씨 알려줘
```

흐름

```
VAD → speech
Endpoint → speech end
Turn detection → user turn finished
```

그 다음

```
AI turn 시작
```

입니다.

# 6️⃣ Turn Detection이 없으면

문제가 생깁니다.

예

```

```

사용자가 아직 말하는 중인데 **AI가 먼저 말합니다.**

그래서 대화가 **부자연스러워집니다.**
